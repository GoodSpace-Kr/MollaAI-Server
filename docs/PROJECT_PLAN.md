# MollaAI Server 기획 문서

작성일: 2026-05-19  
문서 목적: MollaAI 백엔드 서버의 제품 방향, 주요 사용자 흐름, 기능 범위, 시스템 구성, 운영 기준을 팀 내에서 공유하기 위한 기획 문서

## 1. 프로젝트 개요

MollaAI Server는 전화 기반 영어 회화 학습 서비스의 백엔드 API 서버이다. 사용자는 전화 통화를 통해 레벨 테스트 또는 회화 연습을 진행하고, 통화 종료 후 AI가 생성한 피드백 리포트, 영어 레벨 결과, 단어장을 확인할 수 있다.

이 서버는 모바일/웹 클라이언트와 AI 오케스트레이션 서버 사이에서 사용자 인증, 통화 세션 관리, 리포트 조회, 단어 저장, 구독 기반 사용량 관리를 담당한다.

## 2. 서비스 목표

- 전화 통화라는 낮은 진입 장벽으로 영어 회화 연습을 시작할 수 있게 한다.
- 첫 통화는 레벨 테스트로 처리해 사용자의 현재 영어 수준을 파악한다.
- 이후 통화는 개인화된 회화 연습으로 이어지도록 한다.
- 통화 후에는 문법, 어휘, 발화 습관, 발음 메모, 종합 점수, 레벨 결과를 포함한 피드백을 제공한다.
- 채팅 기반 AI 코치를 통해 통화 외 시간에도 학습 맥락을 이어간다.
- 구독과 일일 사용 가능 시간을 관리해 서비스 이용량을 제어한다.

## 3. 주요 사용자

### 3.1 학습자

영어 회화 실력을 점검하고 반복 연습하려는 사용자이다. 전화번호 인증으로 가입하고, 통화 기록과 피드백 리포트, 단어장을 사용한다.

### 3.2 AI 오케스트레이션 서버

실제 통화 연결, 음성 처리, 대화 진행을 담당하는 내부 시스템이다. 통화 시작/종료 시 MollaAI Server의 내부 API를 호출해 세션을 생성하고 종료한다.

### 3.3 운영자/개발팀

서비스 배포, 외부 연동, 장애 대응, 데이터 품질 관리를 담당한다. Swagger, 로그, 데이터베이스, 배포 문서를 기준으로 서버 상태와 API 동작을 확인한다.

## 4. 핵심 가치 제안

- **전화 기반 학습**: 앱 내 복잡한 설정 없이 전화번호와 통화 흐름을 중심으로 학습을 시작한다.
- **AI 피드백 자동화**: 통화 전문을 바탕으로 구조화된 피드백 리포트를 생성한다.
- **학습 자산화**: 단어와 표현을 북마크로 저장해 복습 가능한 자료로 축적한다.
- **구독 기반 제어**: 일일 통화 가능 시간을 기준으로 사용량을 관리한다.

## 5. 현재 기능 범위

### 5.1 인증 및 회원 관리

- 전화번호 기반 SMS 인증번호 발송
- 인증번호 검증 후 Access Token, Refresh Token 발급
- 신규 사용자 여부 반환
- 신규 사용자의 이름 등록
- Access Token 재발급
- 내 정보 조회 및 수정
- 회원 탈퇴 처리

### 5.2 통화 세션 관리

- 내부 API를 통한 통화 세션 시작
- 전화번호 기준 사용자 조회 및 세션 소유자 결정
- 첫 통화는 `level_test`, 이후 통화는 `practice`로 자동 분류
- 통화 시작 시 사용자 상태 스냅샷 저장
- 통화 종료 시 진행 시간 계산 및 상태 변경
- 인증된 사용자의 통화 목록 및 상세 조회

### 5.3 피드백 리포트

- 통화 종료 이벤트 이후 비동기 워커로 리포트 생성
- 통화 전문 기반 AI 피드백 생성
- 한 줄 요약, 문법 교정, 어휘 제안, 습관 분석, 발음 메모, 종합 점수 저장
- 레벨 테스트 통화의 경우 사용자 영어 레벨 자동 갱신
- 리포트 목록 조회 및 세션 단위 상세 조회

### 5.4 단어장

- 단어 또는 표현 저장
- 통화 세션과 연결 가능한 단어 저장
- 저장된 단어 목록 조회
- 본인 소유 단어 삭제

### 5.5 구독 및 사용량 관리

- 현재 활성 구독 조회
- 오늘 잔여 통화 가능 분 반환
- 무료 플랜과 프리미엄 플랜 생성
- 구독 만료 상태 자동 반영
- 활성 구독 중복 생성 방지

### 5.6 검색 연동

- 통화 전문을 Qdrant에 업서트
- 향후 검색과 복습 추천 기능에 활용 가능한 기반 데이터 축적

## 6. 대표 사용자 흐름

### 6.1 최초 사용자 온보딩

1. 사용자가 전화번호를 입력한다.
2. 서버가 SMS 인증번호를 발송한다.
3. 사용자가 인증번호를 검증한다.
4. 신규 사용자라면 이름 등록 화면으로 이동한다.
5. 회원가입 완료 후 첫 통화 또는 레벨 테스트로 진입한다.

### 6.2 첫 통화 레벨 테스트

1. 앱이 사용자 JWT로 세션 시작 API를 호출한다.
2. 서버는 인증된 사용자의 첫 통화 여부를 확인하고 통화 세션을 생성한다.
3. 첫 통화라면 세션 유형을 `level_test`로 저장한다.
4. 서버는 Cloudflare Realtime session을 만들고, 상시 연결된 AI 오케스트레이터 agent WSS로 `join_call`을 전송한다.
5. 앱은 백엔드에 WebRTC offer를 전달하고 Cloudflare Realtime answer를 받아 media connection을 연다.
6. 상시 연결된 AI 서버와 백엔드는 agent control 채널을 기준으로 통화 제어를 진행한다.
7. 통화 종료 시 AI 오케스트레이션 서버가 내부 세션 종료 API를 호출한다.
8. 서버는 통화 전문을 저장하고 비동기 워커를 실행한다.
9. 워커는 AI 리포트를 생성하고 레벨 결과를 사용자 정보에 반영한다.
10. 사용자는 앱에서 리포트와 레벨 결과를 확인한다.

### 6.3 반복 회화 연습

1. 사용자가 다시 통화를 시작한다.
2. 서버는 세션 유형을 `practice`로 저장한다.
3. 서버는 상시 연결된 AI 오케스트레이터 agent WSS로 통화를 배정한다.
4. 통화 종료 후 피드백 리포트를 생성한다.
5. 사용자는 이전 리포트와 단어장을 참고해 복습한다.

## 7. 시스템 구성

### 7.1 애플리케이션

- Java 17
- Spring Boot 3.4.5
- Spring Web
- Spring Data JPA
- Spring Security
- Spring Validation
- Spring WebFlux
- Lombok
- Springdoc OpenAPI

### 7.2 데이터베이스와 저장소

- MySQL: 사용자, 인증, 통화 세션, 리포트, 단어장, 구독 저장
- Qdrant: 통화 전문 벡터 검색 기반 저장

### 7.3 외부 연동

- Naver SENS: SMS 인증번호 발송
- OpenAI API: 리포트 생성, 임베딩
- AI 오케스트레이션 서버: 통화 연결 및 종료 이벤트 전달

### 7.4 배포

- Docker 이미지 기반 배포
- 애플리케이션 포트: `8080`
- Nginx reverse proxy를 통해 외부 `80` 포트에서 내부 `127.0.0.1:8080`으로 전달
- 주요 환경 변수: DB 접속 정보, JWT Secret, Naver SENS 키, OpenAI API 키, Qdrant 접속 정보

| --- | --- |
| User | 전화번호, 이름, 영어 레벨, 회원 상태, Refresh Token 관리 |

| Subscription | 구독 플랜, 일일 사용 가능 시간, 만료 상태 관리 |

## 9. API 범위

### 9.1 공개/클라이언트 API

- `POST /api/v1/auth/send-code`
- `POST /api/v1/auth/verify-code`
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/refresh`
- `GET /api/v1/users/me`
- `PATCH /api/v1/users/me`
- `DELETE /api/v1/users/me`
- `POST /api/v1/sessions/start`
- `GET /api/v1/agents/control` (WebSocket upgrade)
- `GET /api/v1/sessions`
- `GET /api/v1/sessions/{id}`
- `GET /api/v1/reports`
- `GET /api/v1/reports/{sessionId}`
- `GET /api/v1/bookmarks`
- `POST /api/v1/bookmarks`
- `DELETE /api/v1/bookmarks/{id}`
- `GET /api/v1/subscriptions/me`
- `POST /api/v1/subscriptions`

### 9.2 내부 API

- `POST /api/v1/internal/sessions/start`
- `PATCH /api/v1/internal/sessions/{id}/end`

내부 API는 AI 오케스트레이션 서버 전용으로 설계되어 있으며, 외부 공개 API와 별도의 접근 제어 정책이 필요하다. 기존 내부 시작 API는 전화 기반/레거시 연결 경로 호환을 위해 유지할 수 있다.

## 10. 비기능 요구사항

### 10.1 보안

- 클라이언트 API는 JWT 기반 인증을 기본으로 한다.
- Refresh Token은 사용자 단위로 저장하고 만료 시각을 관리한다.
- 앱이 agent control WSS에 접속할 때는 사용자 JWT 대신 백엔드가 발급한 짧은 수명의 agent token을 사용한다.
- 전화번호, 통화 전문, 학습 리포트는 개인정보 및 민감 학습 데이터로 취급한다.
- 내부 API는 운영 환경에서 네트워크 레벨 접근 제어 또는 내부 인증 장치가 필요하다.

### 10.2 안정성

- 통화 종료 후 리포트 생성은 비동기 워커로 처리해 사용자 요청 흐름을 막지 않는다.
- 리포트 생성과 Qdrant 업서트는 단계별 예외 처리를 통해 한 단계 실패가 전체 후처리를 중단시키지 않도록 한다.
- 리포트는 세션당 1개만 생성되도록 중복 저장을 방지한다.

### 10.3 확장성

- 통화 전문 저장과 벡터 저장을 분리해 향후 검색, 추천, 개인화 코칭 기능으로 확장할 수 있다.
- 구독 모델은 무료/프리미엄 플랜과 일일 제한 시간을 기준으로 추가 상품 정책을 수용할 수 있다.
### 10.4 관측성

- 세션 워커 단계별 로그를 통해 리포트 생성과 Qdrant 업서트 상태를 확인한다.
- Swagger UI를 통해 API 명세와 요청/응답 구조를 확인한다.
- 운영 환경에서는 외부 연동 실패율, 워커 처리 시간, 리포트 생성 실패율, 인증 SMS 실패율을 주요 지표로 관리한다.

## 11. 운영 및 배포 기준

- 기본 실행 포트는 `8080`이다.
- Docker Compose는 서버를 `127.0.0.1:8080`에 바인딩하고 Nginx가 외부 요청을 프록시한다.
- 데이터베이스는 MySQL을 사용하며 JPA `ddl-auto`는 현재 `update`로 설정되어 있다.
- 운영 배포 시 환경 변수 누락 여부를 사전에 검증해야 한다.
- OpenAI, Naver SENS, Qdrant 장애 시 사용자 경험에 미치는 영향을 분리해 대응한다.

## 12. MVP 완료 기준

- 전화번호 인증으로 사용자가 가입/로그인할 수 있다.
- 첫 통화가 레벨 테스트로 저장되고, 이후 통화가 연습 세션으로 저장된다.
- 통화 종료 후 피드백 리포트가 비동기로 생성된다.
- 사용자가 앱에서 통화 목록, 리포트 목록, 리포트 상세를 조회할 수 있다.
- 사용자가 단어를 저장하고 삭제할 수 있다.
- 구독 상태와 일일 잔여 사용량을 조회할 수 있다.
