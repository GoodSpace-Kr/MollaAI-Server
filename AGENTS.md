# AGENTS.md

이 파일은 MollaAI Server에서 작업하는 AI 코딩 에이전트와 사람 개발자가 작업 전에 읽는 공통 운영 지침이다. 목표는 빠르게 고치는 것이 아니라, 작은 변경을 정확히 만들고 실제 증거로 확인하는 것이다.

## 1. 프로젝트 컨텍스트

- 프로젝트: MollaAI Server
- 성격: 전화 기반 영어 회화 학습 서비스의 Spring Boot 백엔드
- 주요 기능: SMS 인증, JWT 로그인, 통화 세션, AI 피드백 리포트, AI 코치 채팅, 단어장, 구독/사용량, 사용자 메모리, Qdrant 연동
- 기획 문서: `docs/PROJECT_PLAN.md`
- 배포 문서: `docs/deploy/README.md`
- 기본 런타임: Java 17, Spring Boot 3.4.x, Gradle
- 주요 외부 연동: MySQL, Naver SENS, OpenAI API, Qdrant, AI 오케스트레이션 서버
- 중요 변경 기록: `memories.md`

## 2. 작업 시작 전 체크리스트

작업을 시작하기 전에 다음 순서로 현재 상태를 확인한다.

1. `AGENTS.md`를 읽는다.
2. `memories.md`를 읽어 최근 중요한 변경사항을 확인한다.
3. 요청이 제품/도메인에 영향을 준다면 `docs/PROJECT_PLAN.md`를 확인한다.
4. 변경 대상 주변 코드와 테스트를 먼저 읽는다.
5. `git status --short`로 미추적/수정 파일을 확인한다.
6. 사용자가 만든 변경을 되돌리지 않는다.
7. 요구사항이 불명확해도 코드에서 합리적으로 확인 가능한 것은 직접 확인한다.
8. 확인할 수 없고 잘못 가정하면 위험한 내용만 사용자에게 질문한다.

## 3. 에이전트 운영 루프

널리 쓰이는 에이전트 하네스 패턴인 Observe, Plan, Act, Verify, Report 루프를 따른다.

- **Observe**: 파일, 테스트, 문서, 로그를 읽어 실제 상태를 확인한다.
- **Plan**: 변경 범위를 작게 나누고, 가장 위험한 가정부터 제거한다.
- **Act**: 필요한 파일만 수정한다. 관련 없는 리팩터링은 하지 않는다.
- **Verify**: 변경을 증명하는 명령을 실행하고 출력과 종료 코드를 확인한다.
- **Report**: 무엇을 바꿨고 무엇을 검증했는지 짧고 구체적으로 남긴다.

작업 중 예상과 다른 코드나 실패를 만나면 바로 수정하지 말고 원인을 먼저 좁힌다. 증상, 재현 조건, 관련 코드, 최소 수정, 재검증 순서로 접근한다.

## 4. 하네스 엔지니어링 원칙

### 4.1 명시적 컨텍스트

- 작업 지시, 관련 파일, 제약 조건, 완료 기준을 분리해서 이해한다.
- 추측한 내용은 추측으로 표시하고, 코드나 문서로 확인된 내용과 섞지 않는다.
- API, DTO, 엔티티, 서비스 계층의 기존 네이밍과 책임 경계를 우선한다.

### 4.2 작은 변경 단위

- 한 번의 작업은 하나의 사용자 의도에 집중한다.
- 수정은 요청과 직접 관련된 파일로 제한한다.
- 대규모 정리, 포맷 변경, 구조 변경은 명시적 요청이 있을 때만 한다.
- 새 추상화는 중복 제거, 책임 분리, 테스트 용이성 중 하나 이상을 분명히 개선할 때만 추가한다.

### 4.3 검증 가능한 출력

- “완료”, “통과”, “수정됨” 같은 표현은 검증 명령을 실행한 뒤에만 사용한다.
- 테스트를 실행하지 못했다면 그 이유를 명확히 말한다.
- 문서 변경은 링크, 목차, placeholder, 오탈자, 프로젝트 사실관계 위주로 검증한다.
- 코드 변경은 가능한 가장 가까운 테스트부터 실행하고, 필요하면 더 넓은 테스트로 확장한다.

### 4.4 안전한 도구 사용

- 검색은 `rg` 또는 `rg --files`를 우선한다.
- 단순 파일 확인은 `sed`, `head`, `tail`, `wc`, `git diff`, `git status`를 사용한다.
- 수동 파일 편집은 패치 기반으로 수행한다.
- 파괴적 명령은 사용자가 명시적으로 요청하지 않는 한 실행하지 않는다.
- 비밀값, 토큰, 운영 DB 접속 정보는 출력하거나 문서에 적지 않는다.

### 4.5 테스트 우선 사고

- 버그 수정은 먼저 실패 조건을 이해하고, 가능하면 회귀 테스트를 추가한다.
- 새 기능은 성공 경로뿐 아니라 인증, 권한, 검증 실패, 외부 연동 실패를 고려한다.
- 테스트가 없거나 작성이 과한 작은 변경이라면 최소한 빌드 또는 관련 계층 검증 명령을 실행한다.

## 5. Java/Spring Boot 작업 규칙

- Java 17 문법과 현재 코드 스타일을 따른다.
- 컨트롤러는 요청/응답 DTO와 인증 사용자 추출에 집중한다.
- 서비스는 비즈니스 규칙과 트랜잭션 경계를 담당한다.
- 엔티티는 상태 변경 메서드를 통해 도메인 상태를 바꾼다.
- 예외는 기존 `*Exception`, `ErrorCode`, `GlobalExceptionHandler` 패턴을 따른다.
- API 응답은 기존 `ApiResponse` 포맷을 유지한다.
- 인증이 필요한 API는 SecurityContext에서 현재 사용자 ID를 가져오는 기존 방식을 따른다.
- 내부 API와 클라이언트 API의 책임과 보안 가정을 섞지 않는다.
- 외부 API 호출 실패는 사용자 경험과 후처리 흐름에 미치는 영향을 분리해서 처리한다.

## 6. 데이터와 도메인 주의사항

- 전화번호, 통화 전문, 학습 리포트, 채팅 기록은 민감 데이터로 취급한다.
- 로그에 개인정보, 인증번호, 토큰, 통화 전문 전체를 남기지 않는다.
- `CallSession`의 `level_test`/`practice`, `in_progress`/`completed`/`failed` 같은 문자열 상태는 기존 의미를 보존한다.
- 리포트는 세션당 하나라는 제약을 유지한다.
- 구독과 사용량 변경은 과금 또는 접근 제어로 이어질 수 있으므로 테스트와 경계 조건을 더 꼼꼼히 확인한다.
- 사용자 탈퇴, 토큰 만료, 권한 없는 리소스 접근은 데이터 노출 가능성을 기준으로 검토한다.

## 7. 검증 명령 가이드

상황에 맞게 가장 좁은 검증부터 실행한다.

```bash
./gradlew test
```

전체 테스트가 무겁거나 실패 원인이 외부 환경이라면, 가능한 더 좁은 테스트 태스크나 컴파일 검증을 사용한다.

```bash
./gradlew testClasses
```

문서만 바꾼 경우에는 다음을 확인한다.

```bash
git diff -- AGENTS.md docs/PROJECT_PLAN.md
rg -n "T[O]DO|T[B]D|FIX[M]E|작성[[:space:]]+예정" AGENTS.md docs/PROJECT_PLAN.md
```

검증이 환경 변수, MySQL, Qdrant, 외부 API 키 때문에 실패할 수 있다면 실패 지점을 그대로 기록하고, 로컬에서 확인 가능한 대체 검증을 수행한다.

## 8. 커밋 전 기준

모든 작업은 변경 파일을 `git add`로 스테이징하고 `git commit`까지 완료하는 것을 기본 원칙으로 한다. 사용자가 명시적으로 커밋하지 말라고 했거나, 검증 실패/요구사항 미확정/충돌 등으로 커밋하면 안 되는 상태라면 커밋하지 않고 이유를 보고한다.

커밋 전에는 다음을 확인한다.

- `git status --short`로 변경 파일을 확인한다.
- 내 작업과 무관한 변경은 스테이징하지 않는다.
- 새 파일이 의도한 위치에 있는지 확인한다.
- 중요한 변경사항이 있으면 `memories.md`에 기록했는지 확인한다.
- 테스트 또는 대체 검증 결과를 기록한다.
- 사용자 요청 없이 브랜치 전환, 리셋, 강제 푸시를 하지 않는다.

커밋 메시지는 전통적인 Conventional Commit 타입을 사용한다.

- `feat`: 사용자 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 변경
- `test`: 테스트 추가 또는 수정
- `refactor`: 동작 변경 없는 구조 개선
- `chore`: 빌드, 설정, 의존성, 운영 보조 작업
- `ci`: CI/CD 변경

제목은 명확하고 현재형으로 작성한다. 예: `docs: add project agent and operations docs`, `fix: prevent duplicate feedback report creation`.

## 9. memories.md 기록 규칙

환경변수, 엔드포인트, 메인 로직, 업데이트 내용처럼 운영이나 제품 동작에 영향을 주는 변경사항은 반드시 `memories.md`에 기록한다.

기록이 필요한 대표 사례:

- 환경변수 추가, 삭제, 이름 변경, 의미 변경
- API 엔드포인트 추가, 삭제, 경로 변경, 요청/응답 형식 변경
- 인증, 권한, 보안 정책 변경
- 통화 세션, 리포트 생성, 사용자 메모리, 구독/사용량 계산 같은 핵심 로직 변경
- OpenAI, Naver SENS, Qdrant, MySQL, AWS, GitHub Actions, Docker 연동 변경
- 배포 방식, 포트, 보안그룹, Nginx, Docker 이미지 변경
- 데이터베이스 스키마 또는 마이그레이션 변경
- 운영 장애 대응, 임시 우회, 롤백, hotfix

기록할 때는 실제 secret 값, 개인키, DB 비밀번호, API key, 인증번호, 토큰, 통화 전문 원문을 남기지 않는다. 새 기록은 `memories.md`의 가장 위에 추가한다.

## 10. 코드 리뷰 관점

리뷰를 요청받으면 칭찬보다 위험을 먼저 본다.

- 실제 버그 가능성
- 인증/권한 누락
- 데이터 무결성 문제
- 트랜잭션 경계 문제
- 외부 연동 실패 처리
- 테스트 누락
- 운영 환경에서만 발생할 수 있는 설정 문제

문제가 없으면 “발견한 이슈 없음”이라고 말하고, 남은 리스크나 실행하지 못한 검증만 짧게 남긴다.

## 11. 응답 방식

- 한국어로 간결하게 보고한다.
- 변경한 파일과 검증한 명령을 명시한다.
- 실패한 검증은 숨기지 않는다.
- 불확실한 내용은 단정하지 않는다.
- 사용자가 다음에 바로 판단할 수 있도록 핵심만 남긴다.

## 12. AGENTS.md 변경 규칙

이 파일은 에이전트 작업 품질을 좌우하는 하네스 문서이므로 신중하게 변경한다.

- 프로젝트 구조, 검증 명령, 운영 정책이 바뀌면 함께 갱신한다.
- 특정 작업자의 취향보다 반복 가능한 품질 기준을 우선한다.
- 너무 세부적인 일회성 지시는 넣지 않는다.
- 새 규칙은 “언제 적용되는지”와 “어떻게 확인하는지”가 분명해야 한다.
